Libro Blanco de Seguridad de WordPress

Visión de conjunto

Este documento es un análisis y y explicación del desarrollo de software del core de WordPress y susprocesos de seguridad relacionados, al igual que un examen de la seguridad inherente construida directamente en el software. Las personas que toman decisión y que evaluan WordPress como un sistema de gestión de contenidos o como un marco de aplicaciones web deberían usar este documento en su análisis y toma de decisión, y para los desarrolladores deberían referirse al mismo para familiarizarse con los componentes de seguridad y las mejores prácticas de software.

La información de este documento está actualizada para la última versión estable del software, WordPress 4.1 en el momento de su publicación, pero debería ser considerado relevante también para las versiones más recientes del software ya que la compatibilidad con versiones anteriores es un enfoque fuerte en el equipo sw desarrollo de WordPress. Medidas de seguridad específicas y otros cambios se anotarán a medida que se ańadan al core del software en versiones específicas. Se recomienda encarecidamente que siempre se ejecute la última versión estable de WordPress para asegurarse de tener la experiencia más segura posible.

Sumario Ejecutivo

WordPress es un sistema de gestión de contenidos dinámico de fuentes abiertas que se usa para hacer funcionar millones de sitios web, aplicaciones web y blogs. En la actualidad es usado por más del 23% de 10 millones de sitios web en Internet. La usabilidad, extensibilidad y la madurez de la comunidad de desarrollo hace que sea una elección popular y segura para sitios web de todos los tamańos.

Desde su concepción de 2003, WordPress ha sufrido continuos cambios para endurecerlo de manera que su core pueda gestionar y mititgar las amenazas de seguridad más comunes, incluyendo la lista del Top 10 identificado por El Proyecto de Seguridad para Aplicaciones Web Abiertas (OWASP) como las vulnerabilidades de seguridad más frecuentes, que se discuten en el presente documento.

El equipo de seguridad de WordPress, en colaboración con el equipo lider del Core de WordPress y respaldados por la comunidad global de WordPress, trabaja para identificar y resolver incidencias de seguridad en el core de software disponible para la distribución e instalación en WordPress.org, y también para recomendar y documentar las mejores prácticas en seguridad para plugins de terceras partes y para autores de temas de terceros.

Los desarrolladores de sitios web y los administradores deberían prestar especial atención al uso correcto de las APIs del core y la configuración del servidor que han sido la fuente de vulnerabilidades frecuentes, y también deben asegurarse que todos los usuarios emplean claves fuertes para acceder a WordPress.

Una visión general de WordPress

WordPress es un sistema de gestió de contenidos de fuentes abiertas y gratuito (CMS). Es el CMS m´s extensamente usado en el mundo y funciona en m´ del 23% de los 10 millones de sitios web más vistos1, ofreciendo una cuota de mercado estimada del 60% de todos los sitios que usan un CMS.

WordPress está bajo la Licencia Publica General (GPLv2 o posterior) que proporciona 4 libertades fundamentales, y que puede considerarse como la "carta de derechos" de WordPress

  1. La libertad de ejecutar el programa, para cualquier propósito.
  2. La libertad de estudiar como funciona el programa, y cambiarlo para que haga lo que tu deseas.
  3. La libertad de redistribuirlo.
  4. La libertad de distribuir copias de tus versiones modificadas a otros.

El equipo lider del core de WordPress

El proyecto WordPress es una meritocracia, The WordPress project is a meritocracy, dirigido por un equipo lider central, y conducido por su co-creador y desarrollador principal, Matt Mullenweg. El equipo gobierna todos los aspectos del proyecto, incluyendo el desarrollo del core, WordPress.org, y las iniciativas de la comunidad.

El equipo que lidera el Core está formado por Matt Mullenweg, 5 desarrolladores lider, y más de una docena de desarrolladores del core con acceso permanente a commit. Estos desarrolladores tienen autoridad final en decisiones técnicas, y conducen las discusiones técnicas y los esfuerzos de implementación.

WordPress tiene un número de desarrolladores contribuyendo. Algunos de estos son antiguos o actuales commiters, y algunos son probablemente futuros commiters. Estos desarrolladores que contribuyen son contribuidores veteranos y de confianza de WordPress que se han ganado un gran respeto entre sus homologos. Segú necesidades, WordPress tiene committers invitados, que son individuos a los que se concede acceso para commit, en algunas ocasiones para componentes específicos o con criterios temporales o de prueba.

Los desarrolladores del core y los contribuidores guían de manera principal el desarrollo. En cada versión, cientos de desarrolladores contribuyen al código de WordPress. Estos contribuidores del core son voluntarios que contribuyen al código fuente base de alguna manera.

El ciclo de liberación de versión de WordPress

Cada ciclo de liberación de WordPress es conducido por uno o m´ desarrolladores del core de WordPress. Un ciclo de liberación generalmente dura sobre los 4 meses desde la reunión de alcance inicial hasta el lanzamiento de la versión.

Un ciclo de liberación sigue el siguiente patrón2:

  • Fase 1: Planificación y aseguramiento de la lí del equipo. Esto se hace en la sala de chat #core en Slack. La liberación discute características de la siguiente versión de WordPress. Los contribuidores de WordPress se implican con la mencionada discusión. Las iniciativas de la liberación identificaran las iniciativas del equipo para cada una de las características.
  • Fase 2: Inicio del desarrollo. Los lideres de equipo montan equipos y trabajan en sus funciones o características asignadas. Se programan chats periodicos para asegurarse que el desarrollo sigue progresando.
  • Fase 3: Beta. Se liberan las Betas, y los a los beta-testers se les pide que empiecen a informar de fallos. No se realizan má commits para nuevas mejoras y no se llevan a cabo solicitud de funcionalidades a partir de esta fase. A los autores de terceras partes de plugins y temas se les anima a que prueban su cóodigo contra los cambios que se está realizando.
  • Fase 4: Candidato a liberació. Hay una cadena de congelació para cadenas traducibles a partir de este punto. El trabajo se centra en regresiones e incidencias bloquantes solamente.
  • Fase 5: Lanzamiento. Se lanza la versión de WordPress y se hace disponible en el administrador de WordPress para actualizaciones.

Numeración de Versiony liberaciones de seguridad

Una versió principal de WordPress se dicta por las 2 primeras secuencias. Por ejemplo, 3.5 es una versió principal, al igual que 3.6, 3.7, o 4.0. No hay un "WordPress 3" o "WordPress 4" y cada liberaci&ocute;n o versión principal es referida por su numeración, i.e., "WordPress 3.9."

La liberaciones más importantes pueden ańadir nuevas características y APIs de desarrollo. Aunque lo tipico en el mundo del software, una versión "principal" significa que se pueda romper la compatibilidad con versiones precedentes, WordPress se esfuerza en no romper la compatiblidad con versiones previas. La compatibilidad hacia atras es una de las filosofías más importante del proyecto, con el objetivo de hacer las actualizaciones más fáciles para los usuarios y los desarrolladores por igual.

Una versión menor de WordPress se indica por la tercera secuencia numérica. La versión 3.5.1 es una versión menor, al igual que la 3.4.23. Una versión menor se reserva para arreglos de vulnerabilidades de seguridad y abordar errores críticos solamente. Ya que se liberan nuevas versiones de WordPress con frecuencia - el objetivo es que cada 4-5 meses haya una liberación mayor, y las liberaciones menores se lleven a cabo según necesidad- hay solamente necesidad de liberaciones mayores y menores.

Compatibilidad con versiones precedentes

El proyecto WordPress tiene un fuerte compromiso con la compatibilidad con versiones precedentes. Este compromiso significa que los temas, plugins y el código personalizado continua funcionando cuando el core de WordPress se actualiza, animando a los propietarios del sitio a mantener las versiones del WordPress actualizadas a la última versión segura.

WordPress y la Seguridad

El equipo de seguridad de WordPress

El equipo de seguridad de WordPress está formado por aproximadamente 25 expertos incluyendo los desarrolladores principales y los investigadores de seguridad — aproximadamente la mitad son empleado de Automattic (los creadores de WordPress.com, la plataforma más antigua y grande de alojamiento de WordPress en la web), y el otro conjunto trabajan en el campo de la seguridad web. El equipo consulta con investigadores en seguridad bien conocidos y de confianza y con empresas de alojamiento web.3.

El equipo de seguridad de WordPress a menudo colabora con otros equipos de seguridad The WordPress Security Team often collaborates with other security teams para abordar incidencias en dependencias comunes, tales como resolver la vulnerabilidad en el intérprete de PHP XML, usado por la API de XML-RPC API que se incluye con WordPress, en WordPress 3.9.24. Esta resolución de la vulnerabilidad fue el resultado de un esfuerzo conjunto entre los equipos de seguridad de WordPress y Drupal.

Riesgos de seguridad en WordPress,Proceso e Historia

El equipo de seguridad de WordPress cree en la revelació responsable alertando al equipo de seguridad de manera inmediata ante cualquier vulnerabilidad potencial. Las vulnerabilidades de seguridad potenciales se pueden indicar al equipo de seguridad directamente a través de la dirección de email: security@wordpress.org5. El equipo de seguridad se comunica entre si via una lista privada de correo, y trabaja en un entorno seguro y privado con Trac para seguimiento de incidencias, pruebas y arreglos de fallos y problemas de seguridad.

Cada informe de seguridad es reconocido tras su recepción, y el equipo trabaja para verificar la vulnerabilidad y determinar su grado de severidad. Si se confirma, el equipo de seguridad planifica el parche para solventar el problema que puede ser confirmado para la siguiente versión del software de WordPress o que puede ser subido de inmediato como una versión de seguridad, dependiendo de la severidad de la incidencia.

Para una liberación inmediata de seguridad, se publica un aviso por parte del equipo de seguridad al sitio de noticias de WordPress.org6anunciando la liberación y detallando los cambios. Se da crédito para revelación responsable de una vulnerabilidad en los avisos para animar y reforzar el informe responsable continuado en el futuro.

Los Administradores del software de WordPress ven una notificación en el panel de su sitio web para actualizar cuando hay una versión disponible, y siguiendo el manual de actualizació los usuarios son redirigidos a la pantalla de Acerca de WordPress que detalla los cambios. Si los administradores tienen habilitadas las actualizaciones automáticas en segundo plano, recibirán un email después de que la actualización se haya completado.

Actualizaciones automáticas en segundo plano para versiones de seguridad

Empezando con la versión 3.7, WordPress introdujo actualizaciones automáticas en segundo plano para todas las versiones menores7, tales como la 3.7.1 y 3.7.2. El equipo de seguridad de WordPress puede identificar, arreglar, y promover mejoras de seguridad automáticas para WordPress sin que el propietario del sitio necesite hacer algo por su parte, y la actualización de seguridad se instalará automáticamente.

Cuando una actualización de seguridad se propone para la versión estable vigente de WordPress, el equipo principal realizará también dicha actualización de seguridad para todas las versiones que sean capaces de recibir actualizaciones en segundo plano (desde WordPress 3.7), de manera que versiones más antiguas pero recientes de WordPress reciban estas mejoras de seguridad.

Los propietarios individuales de sitios web pueden optar por quitar la actualizaciones automáticas en segundo plano a través de un simple cambio de su fichero de configuración, pero se recomienda mantener está funcionalidad de manera vehemente por parte del equipo principal, al igual que ejecutar la versión estable más reciente de WordPress.

El Top 10 de OWASP 2013

Proyecto Abierto de Seguridad de Aplicaciones Web(OWASP) es una comunidad online dedicada a la seguridad de aplicaciones web. La lista de los 10 principales de OWASP8 se centra en la identicación de los riesgos de seguridad más importantes en aplicaciones para una amplia gama de organizaciones. Los 10 elementos más importantes son seleccionados y priorizados en combinación con la estimación consensuada sobre explotabilidad, detectabilidad e impacto.

Las siguientes secciones discuten las APIs, recursos, y politicas que usa WordPress para endurecer el núcleo de software y los plugins y temas de terceras partes contra estos riesgos potenciales.

A1 - Inyección

Hay un conjunto de funciones y APIs disponibles en WordPress para ayudar a los desarrolladores en que se aseguren que no se pueda inyectar código no autorizado, y ayudarles a validar y sanitizar los datos. Las mejores prácticas y documentación están disponibles9 sobre como usar estás APIs para proteger, validar o sanitizar la entrada y salida de datos en HTML, URLs, cabeceras HTTP, y cuando se interacciona con la base de datos o el sistema de ficheros. Los Administradores pueden además restringir los tipos de fichero que se pueden subir mediante filtros.

A2 - Rotura de autenticación y gestión de sesiones.

A3 - Secuencias de órdenes en sitios cruzados (XSS)

WordPress proporciona una rang de funciones que pueden ayudar a asegurarse que los datos suministrados por el usuario son seguros10. Los usuarios de confianza, esto es, los administradores y editores de una instalación sencilla de WordPress, y los administradores de sitios Multisite, pueden publicar HTML o JavaScript sin filtrar si lo necesitan, tales como dentro de un post o una página. Los usuarios que no son de confianza y el contenido enviado por los usuarios se filtra, por defecto, para quitar las entidades peligrosas, usando la biblioteca KSES mediante la funciónwp_kses.

A4 - Referencia a objectos directos inseguros

WordPress a menudo proporciona referencias a objetos directos, tales como identificadores numéricos únicos de cuentas de usuario o contenido disponible en la URL o campos de formularios. Mientras estos identificadores revelan información directa del sistema, el rico sistema de control de acceso y permisos WordPress previene las solicitudes no autorizadas.

A5 - Mala configuración de Seguridad

La mayor parte de operaciones de configuración de seguridad están limitadas a un solo administrador autorizado. La configuración por defecto de WordPress es evaluada de manera continuada a nivel del equipo principal, y el equipo principal de WordPress proporciona documentación y las mejores prácticas para reforzar la seguridad de la configuración del servidor, que ejecuta un sitio WordPress11.

A6 - Revelación de datos sensibles

A7 - Funciones de nivel de control de acceso ausentes

WordPress comprueba la adecuada autorización y permisos en cualquier función de solicitud de nivel de acceso antes de que la acción se lleve a cabo. El acceso o la visualización de URLs administrativas, menús, y páginas sin la adecuada autenticación está ligada e integrada con el sistema de autenticación para prevenir acceso a usuarios no autorizados.

A8 - Falsificación de petición en sitios cruzados (CSRF)

A9 - Using Components with Known Vulnerabilities

The WordPress core team closely monitors the few included libraries and frameworks WordPress integrates with for core functionality. In the past the core team has made contributions to several third-party components to make them more secure, such as the update to fix a cross-site vulnerability in TinyMCE in WordPress 3.5.214.

If necessary, the core team may decide to fork or replace critical external components, such as when the SWFUpload library was officially replaced by the Plupload in 3.5.2, and a secure fork of SWFUpload was made available by the security team15 for those plugins who continued to use SWFUpload in the short-term.

A10 - Unvalidated Redirects and Forwards

WordPress' internal access control and authentication system will protect against attempts to direct users to unwanted destinations or automatic redirects. This functionality is also made available to plugin developers via an API, wp_safe_redirect()16.

Further Security Risks and Concerns

XXE (XML eXternal Entity) processing attacks

When processing XML, WordPress disables the loading of custom XML entities to prevent both External Entity and Entity Expansion attacks. Beyond PHP's core functionality, WordPress does not provide additional secure XML processing API for plugin authors.

SSRF (Server Side Request Forgery) Attacks

HTTP requests issued by WordPress are filtered to prevent access to loopback and private IP addresses. Additionally, access is only allowed to certain standard HTTP ports.

WordPress Plugin and Theme Security

The Default Theme

WordPress requires a theme to be enabled to render content visible on the frontend. The default theme which ships with core WordPress (currently "Twenty Fifteen") has been vigorously reviewed and tested for security reasons by both the team of theme developers plus the core development team.

The default theme can serve as a starting point for custom theme development, and site developers can create a child theme which includes some customization but falls back on the default theme for most functionality and security. The default theme can be easily removed by an administrator if not needed.

WordPress.org Theme and Plugin Repositories

There are approximately 30,000+ plugins and 2,000+ themes listed on the WordPress.org site. These themes and plugins are submitted for inclusion and are manually reviewed by volunteers before making them available on the repository.

Inclusion of plugins and themes in the repository is not a guarantee that they are free from security vulnerabilities. Guidelines are provided for plugin authors to consult prior to submission for inclusion in the repository17, and extensive documentation about how to do WordPress theme development18 is provided on the WordPress.org site.

Each plugin and theme has the ability to be continually developed by the plugin or theme owner, and any subsequent fixes or feature development can be uploaded to the repository and made available to users with that plugin or theme installed with a description of that change. Site administrators are notified of plugins which need to be updated via their administration dashboard.

When a plugin vulnerability is discovered by the WordPress Security Team, they contact the plugin author and work together to fix and release a secure version of the plugin. If there is a lack of response from the plugin author or if the vulnerability is severe, the plugin/theme is pulled from the public directory, and in some cases, fixed and updated directly by the Security Team.

The Theme Review Team

The Theme Review Team is a group of volunteers, led by key and established members of the WordPress community, who review and approve themes submitted to be included in the official WordPress Theme directory. The Theme Review Team maintains the official Theme Review Guidelines19, the Theme Unit Test Data20, and the Theme Check Plugin21, and attempts to engage and educate the WordPress Theme developer community regarding development best practices. Inclusion in the group is moderated by core committers of the WordPress development team.

The Role of the Hosting Provider in WordPress Security

WordPress can be installed on a multitude of platforms. Though WordPress core software provides many provisions for operating a secure web application, which were covered in this document, the configuration of the operating system and the underlying web server hosting the software is equally important to keep the WordPress applications secure.

A Note about WordPress.com and WordPress security

WordPress.com is the largest WordPress installation in the world, and is owned and managed by Automattic, Inc., which was founded by Matt Mullenweg, the WordPress project co-creator. WordPress.com runs on the core WordPress software, and has its own security processes, risks, and solutions22. This document refers to security regarding the self-hosted, downloadable open source WordPress software available from WordPress.org and installable on any server in the world.

Appendix

Core WordPress APIs

The WordPress Core Application Programming Interface (API) is comprised of several individual APIs23, each one covering the functions involved in, and use of, a given set of functionality. Together, these form the project interface which allows plugins and themes to interact with, alter, and extend WordPress core functionality safely and securely.

While each WordPress API provides best practices and standardized ways to interact with and extend WordPress core software, the following WordPress APIs are the most pertinent to enforcing and hardening WordPress security:

Database API

The Database API24, added in WordPress 0.71, provides the correct method for accessing data as named values which are stored in the database layer.

Filesystem API

The Filesystem API25, added in WordPress 2.626, was originally created for WordPress' own automatic updates feature. The Filesystem API abstracts out the functionality needed for reading and writing local files to the filesystem to be done securely, on a variety of host types.

It does this through the WP_Filesystem_Base class, and several subclasses which implement different ways of connecting to the local filesystem, depending on individual host support. Any theme or plugin that needs to write files locally should do so using the WP_Filesystem family of classes.

HTTP API

The HTTP API27, added in WordPress 2.728 and extended further in WordPress 2.8, standardizes the HTTP requests for WordPress. The API handles cookies, gzip encoding and decoding, chunk decoding (if HTTP 1.1), and various other HTTP protocol implementations. The API standardizes requests, tests each method prior to sending, and, based on your server configuration, uses the appropriate method to make the request.

Permissions and current user API

The permissions and current user API29 is a set of functions which will help verify the current user's permissions and authority to perform any task or operation being requested, and can protect further against unauthorized users accessing or performing functions beyond their permitted capabilities.
